Devops 監控

特定軟體版本找出問題並收集相關資訊。工具包括 New Relic、Datadog、Grafana、Wireshark、Splunk、Nagios 或 Slack。

工具簡介

  1. AWS CloudWatch 目的：監控和管理 AWS 資源和應用程式。 主要特點：
  • 提供 AWS 服務的即時監控。
  • 允許根據指標閾值設定警報並自動響應。
  • 與其他 AWS 服務無縫整合。
  1. Prometheus 用途：開源系統監控和警報工具包。 主要特點：
  • 高效率收集和儲存時間序列資料。
  • 提供用於分析指標的強大查詢語言 (PromQL)。
  • 非常適合監控雲端原生應用程式，尤其是使用微服務的應用程式。
  1. Alert Managing 用途：處理 Prometheus 發送的警報並管理通知。 主要特點：
  • 根據定義的規則將警報路由到正確的接收者。
  • 與電子郵件、Slack 和 PagerDuty 等各種通知平台整合。
  1. Grafana 目的：一個用於監控和可觀察性的開源平台。 主要特點：
  • 允許建立可自訂的互動式儀表板。
  • 支援廣泛的資料來源，包括 Prometheus、Elasticsearch 等。
  • 提供警報功能，能夠在儀表板上可視化警報。
  1. Elasticsearch 用途：處理搜尋的引擎，不僅是一個強大的搜尋引擎，也是一個分散式資料庫 Elastic Stack ⇒ Elasticsearch、Kibana、Logstash、Beats

  • Kibana 當作 Elasticsearch 的 Dashboard(視覺化的圖表)

  • Logstash 處理應用程式的 Log 並把它送到 Elasticsearch，Logstash 本身把接受到的資料視作   事件，Logstash 處理後並送到一個或多個地方。 像是 Elasticsearch, Kafka, HTTP Endpoint 都可以作為目的地發送。

  • Beats Logstash 的前一個關卡，負責搜集資料 Beat 有各種不同的款式，像是 Filebeat 就是專門搜集 Web server log, database log 等等，也有像是 Metricbeat 專門來搜集系統的 CPU 以及記憶體用量

    6.kibana 用途： Elasticsearch 的可視化和資料探索工具。 主要特點：

  • 專注於日誌和時間序列分析。

  • 使用戶能夠為 Elasticsearch 資料建立視覺化和儀表板。

  • 包括搜尋、查看 Elasticsearch 中儲存的資料以及與之互動的功能。

    7.Telegraf

  • 資料收集器

  • InfluxDB 即時分析而構建一個高速讀寫資料庫

Telegraf + InfluxDB

Telegraf 就像 Node Exporter 一樣，是一套裝在每臺機器上、定時監控所有機器的 agent

  • Telegraf 設計上會主動把資料給 Database
  • Telegraf 本身具備眾多 plug-in，只要在設定裡啟動就可以了
  • Telegraf 可以設定你要讀取怎樣的數據
  • 可以用 Telegraf 去讀其他數據，再傳給 Database
  • 可以在 Telegraf 內處理數據後傳給 Database
  • 可以整合與運算(如極值、均值…等等)後再傳給 Database

Prometheus

Prometheus 提供的是一整套監控系統, 包括資料的收集、資料儲存、報警, 甚至是繪圖(只不過很陽春,官方推薦使用 grafana) 。但 Prometheus 也需要 Node_exporter，通過 HTTP 定期從這些目標中以基於文本的格式 Pull 指標，並將收集到的資料儲存在本地 time series db 中。

Prometheus 和 InfluxDB 比較

InfluxDB 只是一個 time series db, 使用它做監控系統的話, 還需要物色資料獲取器,如 telegraf, collectd 等. 甚至連報警模組, 也需要使用同為 Influxdata 公司出的 Kapacitor。

Prometheus 和 InfluxDB 在資料的採集上兩者就選擇了不同的極端, 前者只能 pull, 後者只能 push, 關於 pull 和 push 的對比。(Prometheus 也可以經由 PushGateway 來採集 Push mode 的資料)。

Prometheus 把資料的採集器叫做 exporter, xxx-exporter 運行之後會在機器上佔用一個 port, 等待 Prometheus server 來拉取資料。

InfluxDB 的資料搜集器使用了 telegraf, 官方宣傳外掛程式化驅動, 編譯的時候把很多東西的採集功能包進去壓在一個二進位裡面, 再用設定檔控制外掛程式是否開啟. 功能其實是蠻強大了. 也是 Go 編寫。

Prometheus 與 Telegraf：有什麼區別？

Telegraf Telegraf 的架構是基於插件(像是 node exporter)，可以靈活地從各種來源收集資料、進行處理並將其輸出到不同的目的地。 Prometheus 是一個開源監控和警報工具包, 注重可靠性和可擴展性，尤其是在動態雲端環境

Prometheus 和 Telegraf 都是監控和指標收集領域廣泛使用的工具。雖然它們的用途相似，但它們之間存在關鍵的差異，使得每個工具都獨一無二且適合特定的用例。

  • 資料收集：Prometheus 是一個基於拉取的系統，它透過從目標拉取指標來收集資料。另一方面，Telegraf 是一個多功能代理，支援拉取和推送機制。它可以透過從各種來源提取指標以及將指標推送到不同目的地來收集數據。

  • 可擴展性：Prometheus 設計為高度可擴展，可處理大量資料。它透過聯合架構實現可擴展性，其中多個 Prometheus 伺服器可以聯合在一起。另一方面，Telegraf 是一個輕量級收集器，可以部署在大量機器上，因此也具有高度的可擴展性。

  • 資料處理：Prometheus 自帶查詢語言 PromQL，允許對指標進行高階查詢和處理。使用 PromQL，使用者可以對收集的資料執行聚合、篩選和數學運算。另一方面，Telegraf 更注重資料收集和路由，將資料處理任務留給堆疊中的其他工具。

  • 插件生態系統：Telegraf 擁有豐富的插件生態系統，可輕鬆與各種系統和技術整合。它支援廣泛的輸入和輸出插件，使其靈活且易於擴展。而 Prometheus 則更注重核心的監控和警報功能，官方插件數量有限。

  • 警報：Prometheus 具有內建警報系統，可讓使用者根據收集到的指標定義警報規則。它支援各種通知管道，例如電子郵件、Slack 和 PagerDuty，以便在滿足特定條件時發送警報。另一方面，Telegraf 沒有內建警報系統，而是依賴監控堆疊中的其他工具來實現警報功能。
